# Provenance Guard

A backend service any creative-writing platform can plug into to answer one question about a
piece of submitted text: **was this written by a person, or generated by AI?** — and to do it
*honestly*, communicating uncertainty instead of forcing a binary, showing readers a
plain-language transparency label, and giving creators a way to appeal.

The design goal throughout is **not** to be a perfect AI detector (that is an unsolved
problem) but to be an *honest* one: on a writing platform, wrongly branding a real person's
work as AI is worse than missing an AI submission, so the system is deliberately built to say
"we're not sure" rather than falsely accuse a creator.

Full design rationale lives in [`planning.md`](./planning.md). This README is the canonical
record of what was built and why.

## Setup & run

```bash
python -m venv .venv
source .venv/bin/activate            # Windows: .venv\Scripts\activate
pip install -r requirements.txt

# add your Groq key (never commit this file - it is in .gitignore)
echo "GROQ_API_KEY=your_key_here" > .env

python app.py                        # serves on http://localhost:5000
```

> **macOS note:** port 5000 is used by AirPlay Receiver on recent macOS. If `/submit` hangs
> or returns nothing, either turn AirPlay Receiver off (System Settings → General → AirDrop &
> Handoff) or change the port in `app.py` to 5001.

Quick check, then fire the four calibration inputs with the included helper:

```bash
curl -s http://localhost:5000/                 # {"status":"ok",...}
./test_samples.sh                              # POSTs 4 sample texts + dumps the log
```

The audit log is stored in SQLite at `audit_log.db` (path overridable with the
`PROVENANCE_DB` env var).

## API reference

| Method & path | Accepts | Returns |
|---|---|---|
| `GET /` | — | health check |
| `POST /submit` | `{ "text": str, "creator_id"?: str }` | `content_id`, `attribution`, `ai_likelihood`, `confidence`, `confidence_tier`, per-signal `signals`, and the `label` text |
| `POST /appeal` | `{ "content_id": str, "creator_reasoning": str }` | `appeal_id`, `content_id`, `status: "under_review"`, `logged_at` |
| `GET /log` | optional `?limit=`, `?status=` | `{ "entries": [ ... ] }` structured audit entries |
| `GET /submission/<content_id>` | — | original decision + any appeals, joined (reviewer view) |

Errors: `400` bad/missing input, `404` unknown `content_id`, `429` rate-limited.

*(`text` is the primary body field; `content` is also accepted — see [Spec reflection](#spec-reflection).)*

## Architecture

A submission passes the **rate limiter**, then the **detection pipeline** runs two independent
signals — **stylometry** (structural) and the **Groq LLM classifier** (semantic) — whose
scores feed the **confidence scorer**, which fuses them into an `ai_likelihood`, a
`confidence`, and an attribution. The **label generator** renders a plain-language label, the
**audit logger** (SQLite) records the whole decision, and the API returns structured JSON. On
appeal, the **appeals handler** logs the appeal beside the original decision and flips the
content's status to `under_review` for a human moderator.

The full ASCII diagram of both the submission and appeal flows is in
[`planning.md` → Architecture](./planning.md#1-architecture). Component map:

| File | Role |
|---|---|
| `app.py` | Flask API, routes, rate limiting |
| `signals/stylometry.py` | Signal 1 — structural heuristics |
| `signals/llm.py` | Signal 2 — Groq LLM classifier |
| `scoring.py` | confidence fusion + verdict thresholds |
| `labels.py` | transparency label generator |
| `audit.py` | SQLite structured audit log |

## Detection signals

The pipeline uses **two genuinely independent signals** — one structural, one semantic. They
fail in different ways, and that is the whole point: fusing decorrelated signals is more
informative than either alone, and requiring them to *agree* before a strong call is the core
safety mechanism.

### Signal 1 — Stylometric heuristics (structural)

**What it measures:** surface-form statistics computed in pure Python — sentence-length
variance ("burstiness"), type-token ratio (vocabulary diversity), punctuation density, and an
**AI-cliché/formulaic-connective density** ("furthermore," "in conclusion," "it is important
to note," listicle "first/second/third"). Output is a single `p_ai ∈ [0,1]` plus a
`self_confidence` that drops on short text.

**Why this property:** human writing is bursty and uneven; instruction-tuned LLMs regress
toward the mean — sentences cluster around a comfortable length, punctuation is tidy, and they
lean on a recognizable set of connective clichés. Low variance + high cliché density is a
structural fingerprint of AI text, and it is cheap and deterministic.

**What it misses:** it is blind to *meaning*. It is unreliable on short text (the statistics
become noise), it false-positives on intentionally uniform human forms (formal legal prose,
constrained poetry), and it is trivially gameable by anyone who varies sentence length or
avoids the cliché words. TTR in particular is inflated on short passages and barely
discriminates there — which is exactly why the cliché sub-metric was added (see
[Spec reflection](#spec-reflection)).

### Signal 2 — LLM classifier (semantic)

**What it measures:** a holistic judgment from Groq's `llama-3.3-70b-versatile` of whether the
text *reads* as machine-generated — generic phrasing, unnatural smoothness, hedged structure,
lack of specific lived detail. Returns `p_ai ∈ [0,1]`, a one-sentence rationale, and an
`available` flag.

**Why this property:** LLMs are good at recognizing the register of other LLMs — the even,
slightly hollow "AI voice" — and catch AI text that is *statistically* varied but semantically
generic, which stylometry cannot see.

**What it misses:** it is another AI judging AI, so it is uncalibrated and can be confidently
wrong; it is non-deterministic (the same text scores differently across calls); and it is
prone to **false positives on polished or non-native-English human writing**, which is the
harm we most want to avoid. It can also be prompt-manipulated.

**Why the pair works:** stylometry is fooled by statistically-varied-but-generic AI (the LLM
catches it), and the LLM is fooled by clean human prose (whose natural burstiness stylometry
rewards). Neither alone is trustworthy; together, disagreement is a signal in itself.

**If I were deploying this for real:** I would replace the hand-tuned stylometry thresholds
with values fit on a labeled corpus, add a third signal that is not itself an LLM (e.g. a
perplexity/token-probability model) so the ensemble isn't dominated by "AI judging AI," and
calibrate the LLM's raw `p_ai` against ground truth rather than trusting it as a probability.

## Confidence scoring

The system reports **two numbers** because they answer two different questions:

- **`ai_likelihood` ∈ [0,1]** — which direction, and how strongly, the evidence points to AI.
- **`confidence` ∈ [0,100]** — how much to trust the verdict. 100 = sure, 0 = coin flip.

I decided what `confidence` should *mean to a user* before writing the math: a confidence of
~50 means "we lean this way but wouldn't stake much on it," which the label then softens with
hedged wording rather than a firm verdict.

**Fusion:**

```
ai_likelihood = 0.65 · p_ai_llm + 0.35 · p_ai_stylo
agreement     = direction-aware (1.0 when both signals point the same way relative to 0.5,
                lower only when they genuinely point in opposite directions)
confidence    = |ai_likelihood − 0.5| · 2 · agreement · stylometry_self_confidence · 100
```

Three independent things drive confidence down: a borderline combined score, disagreeing
signals, and too-short input. If the LLM is unavailable, the system degrades to stylometry
alone and hard-caps confidence into the low tier (a single signal is never "high confidence").

**Asymmetric verdict thresholds** — because a false positive (calling a human's work AI) is
worse on a writing platform, `AI` is *hard to earn*:

| Condition | Verdict |
|---|---|
| `ai_likelihood ≥ 0.70` **and** `agreement ≥ 0.5` | `likely_ai` |
| `ai_likelihood ≤ 0.35` | `likely_human` |
| everything else, or `confidence < 35` | `uncertain` |

**Confidence tiers** decide which label wording is shown: `≥ 65` high, `35–64` moderate,
`< 35` low.

### Two example submissions (noticeably different confidence)

Real results from the calibration set (LLM values are representative — live Groq is
non-deterministic, so exact numbers wobble a little between runs):

| Input | stylo `p_ai` | llm `p_ai` | `ai_likelihood` | **confidence** | verdict |
|---|---|---|---|---|---|
| **High-confidence:** casual human ("ok so i finally tried that ramen place…") | 0.15 | 0.05 | 0.08 | **83 (high)** | `likely_human` |
| **Low-confidence:** borderline / lightly-edited AI ("I have been thinking a lot about remote work…") | 0.23 | 0.40 | 0.34 | **32 (low)** | `uncertain` |
| blatant AI listicle ("In today's ever-evolving digital landscape…") | 0.70 | 0.98 | 0.88 | 68 (high) | `likely_ai` |
| formal AI essay ("…transformative paradigm shift…") | 0.53 | 0.80 | 0.71 | 41 (moderate) | `likely_ai` |

The casual-human piece (both signals agree strongly → **83**) and the borderline piece (mixed
signals near the middle → **32**) produce a ~50-point confidence gap and *different labels*,
which is the evidence that the score varies meaningfully rather than being a constant.

### How I validated the scores are meaningful

I ran a deliberately chosen set spanning the range — obvious AI (formal essay, listicle),
obvious human (casual/typo-laden), and two ambiguous cases (formal human academic prose;
lightly-edited AI) — and confirmed: the obvious cases clear the high thresholds, the ambiguous
cases land `uncertain` with confidence < 35, and crucially the **formal-human** passage stays
`uncertain` (confidence 6–8) instead of being flagged AI, even when the LLM leaned AI on it —
the false-positive guard working. I also re-submitted identical text several times to observe
the LLM's non-determinism and confirmed it doesn't flip a high-confidence verdict.

## Transparency label variants

The label is the only thing a non-technical reader sees. It uses plain language (no
"classifier," "logit," or "score vector") and the **wording** changes with confidence, not
just the number. Exact text each variant displays (from `labels.py`; the `%` is filled with
the actual confidence):

**High-confidence AI**

> 🤖 Likely AI-generated. Our checks strongly suggest this text was produced by an AI tool.
> Both how it's written (sentence rhythm and word variety) and how it reads overall point the
> same way. Confidence: high (68%). This is an automated assessment, not a certainty — if you
> wrote this and disagree, you can appeal.

**High-confidence human**

> ✍️ Likely written by a person. Our checks found the natural variety in rhythm and word
> choice we'd expect from a human writer, and nothing that points to AI generation. Confidence:
> high (83%). No automated check is perfect, but this reads as human work.

**Uncertain**

> ❓ We're not sure. Our checks came back mixed or borderline, so we're not putting a label on
> this one. Confidence: low (32%). We'd rather say "we don't know" than risk wrongly calling a
> person's work AI-generated. A human reviewer can take a closer look if the creator requests
> it.

Two additional **moderate-tier** variants soften the verdict for directional-but-not-firm
cases (reachable, but not required):

> 🤖 Leans toward AI-generated. Some signs point to AI authorship, but this isn't a firm call.
> Confidence: moderate (41%). If you wrote this yourself and disagree, you can appeal.

> ✍️ Leans toward human-written. This mostly reads like a person's writing, though we're not
> fully certain. Confidence: moderate (55%). If anything here looks off to you, you can appeal.

All three required variants were verified reachable end-to-end by submitting inputs at
different confidence levels (high-AI 68%, high-human 83%, uncertain 6%/32%).

## Appeals workflow

A creator who believes they were misclassified submits an appeal (the platform passes it on
their behalf, identified by `content_id`). The appeal **must** include `creator_reasoning`;
an empty one is rejected (`400`), and an unknown `content_id` returns `404`.

On receipt the system: (1) writes an **appeal entry** to the audit log linked to the original
decision, snapshotting the contested verdict/confidence; (2) updates the content's status to
**`under_review`**; and (3) returns confirmation. Automated re-classification is intentionally
out of scope — an appeal escalates to a human, it does not silently re-run the detector.
`GET /submission/<id>` gives a moderator the original decision and the appeal in one joined
view.

```bash
curl -s -X POST http://localhost:5000/appeal -H "Content-Type: application/json" \
  -d '{"content_id": "PASTE-ID", "creator_reasoning": "I wrote this myself from personal experience. I am a non-native English speaker and my style may appear more formal than typical."}'
```

Response:

```json
{
  "appeal_id": "8fe31160-45df-41fd-80a5-62b5e2189c5f",
  "content_id": "476d3d00-b32b-46c3-b3ba-71cc4a5e4ef8",
  "creator_reasoning": "I wrote this myself from personal experience; please re-review.",
  "logged_at": "2026-07-01T02:32:54.821Z",
  "message": "Appeal received. This content is now under review by a human moderator.",
  "status": "under_review"
}
```

## Rate limiting

`Flask-Limiter` (in-memory store) is applied to `POST /submit`:

**Chosen limits: `10 per minute` and `100 per hour`, per client IP.**

**Reasoning (tied to realistic writing-platform usage):**
- A genuine creator publishes a handful of pieces in a sitting at most. **10/minute**
  comfortably covers editing-and-resubmitting a draft several times, while being far below
  what a script needs to flood the system — and every submission costs a Groq API call, so
  the limit also protects a real cost.
- **100/hour** is a hard ceiling that stops sustained abuse — e.g. an adversary probing the
  detector repeatedly to reverse-engineer its thresholds — without ever inconveniencing a real
  writer, who will not publish 100 pieces in an hour.

Exceeding a limit returns **HTTP 429** with a JSON body. Evidence from the 12-rapid-request
test (first 10 succeed, rest are limited):

```
200 200 200 200 200 200 200 200 200 200 429 429
```

Sample 429 body:

```json
{
  "error": "rate limit exceeded",
  "detail": "10 per 1 minute",
  "limit": "10 per minute;100 per hour",
  "message": "Too many submissions. Please slow down and try again shortly."
}
```

## Known limitations

**Formal, polished writing by a non-native English speaker is the case this system is most
likely to get wrong — a false positive.** This is not a generic disclaimer; it follows
directly from both signals' failure modes. The **LLM signal** keys on register and reads
"clean, formal, slightly generic" prose as AI-like, so ESL writers who write carefully and
formally score high on `p_ai`. The **stylometry signal** can compound this: formal writing
tends toward uniform sentence length and tidy punctuation, which reads as low burstiness. When
*both* signals lean the same wrong way, they *agree*, so the agreement mechanism doesn't rescue
it. The asymmetric 0.70 AI threshold is the main guard (it takes strong evidence to earn an
`AI` verdict, so these usually land `uncertain` rather than `likely_ai`), and the appeal path
is the backstop — but a sufficiently formal ESL passage can still be misclassified, which is
precisely why the appeal workflow exists and why the label never states a verdict as certainty.

A second, opposite failure: **AI text that has been lightly hand-edited** defeats the system as
a false negative — editing adds the sentence-length variety stylometry rewards and removes the
generic phrasing the LLM keys on, so both signals soften and the result is `uncertain` or
`likely_human`. Perfect detection here is unsolved; the system's honest response is to abstain
and offer appeal, not to guess.

## Spec reflection

**One way the spec guided implementation.** Writing `planning.md` *before* any code — with the
exact fusion formula, weights, thresholds, and the three label variants spelled out — meant
each implementation milestone had a concrete target. When I generated the confidence scorer, I
wasn't inventing scoring logic on the fly; I was implementing §3's formula and checking the
output against the thresholds I had already committed to. The architecture diagram kept the
components decoupled (signals → scorer → labels → audit) so I could build and test one signal
end-to-end before adding the second, exactly as planned.

**One way the implementation diverged, and why.** The plan (§3) defined `agreement` as
`1 − |p_llm − p_stylo|` and set the AI threshold at 0.75 with a 0.60/0.40 LLM/stylometry
weighting. Live testing broke both. First, that agreement formula punished a *neutral*
stylometry score (≈0.5) as if it were disagreement, crushing confidence even when the LLM was
sure — so I changed it to a **direction-aware** measure that only penalizes signals pointing in
genuinely opposite directions. Second, real `llama-3.3-70b` returns `p_ai ≈ 0.8` on clearly-AI
text, not the ~0.9 I had assumed, so the 0.75 threshold left obvious AI prose stuck at
`uncertain`. I recalibrated to LLM weight 0.65 and AI threshold 0.70, and verified the change
surfaces `likely_ai` on AI text while keeping the formal-human case `uncertain`. I also added a
fourth stylometry sub-metric (AI-cliché density) that the plan didn't have, because TTR turned
out to be useless on short text. (Minor: the plan named the request field `content`; I accept
`text` as primary to match the platform's expected contract, and both work.)

## AI usage

I used an AI coding assistant (Claude) throughout, always feeding it my `planning.md` sections
and architecture diagram as the spec. Specific instances:

**1. Generating the stylometry signal + fusion, then overriding the calibration.** I directed
the AI to implement Signal 1 and the confidence scorer from planning §2/§3, giving it the exact
output shape and fusion formula. It produced working code that matched the spec — but when I ran
the milestone's calibration inputs, clearly-AI text scored `uncertain` because TTR is
length-inflated on short passages and couldn't flag uniform AI prose. I **overrode** the design
by adding an AI-cliché-density sub-metric (which the spec didn't include) and re-weighting the
sub-signals, because the spec's original three metrics genuinely couldn't discriminate at short
lengths. I decided the thresholds myself against real output rather than accepting the AI's
initial values.

**2. Generating the confidence scorer, then rewriting the agreement term.** I asked the AI to
implement the fusion math from §3 verbatim, including `agreement = 1 − |p_llm − p_stylo|`. It
did exactly that. Testing showed the formula was wrong in practice — a neutral second signal
tanked confidence on cases where the primary signal was decisive — so I **replaced** it with a
direction-aware agreement function of my own design and documented the divergence. This was a
case where following the spec literally produced bad behavior, and I chose to diverge.

**3. Generating the label generator + `/appeal` endpoint, then verifying against the spec.** I
directed the AI to build `labels.py` from the §4 variant text and the appeal endpoint from the
§5 workflow. I **verified** rather than trusted: I confirmed all three label variants were
reachable by submitting inputs at different confidence levels, and checked that an appeal
actually flips status to `under_review` and lands in the log beside the original decision
before considering it done. The one change I made was storing the reasoning under both
`creator_reasoning` and `appeal_reasoning` keys so it's easy to find in the log.
