# Provenance Guard — planning.md

A backend service any creative-writing platform can plug into to (1) classify whether
submitted text was written by a human or generated by AI, (2) score its confidence in a way
that communicates uncertainty instead of forcing a binary, (3) surface a plain-language
transparency label to readers, and (4) let creators appeal a classification.

This document is the spec. It is written *before* any implementation, and its sections are
the source of truth I feed to AI code-generation tools in Milestones 3–5. Section numbers are
stable so I can reference them in prompts (e.g. "implement the fusion formula in §4").

## 1. Architecture

**Submission flow (narrative).** A creator hits "Publish"; the platform forwards the text to
`POST /submit`. The request passes the **rate limiter**, then the **detection pipeline** runs
both signals — the **stylometry analyzer** (structural) and the **Groq LLM classifier**
(semantic) — whose scores feed the **confidence scorer**, which fuses them into an
`ai_likelihood`, a `confidence`, and a `verdict`. The **label generator** turns that into
plain-language text, the **audit logger** (SQLite) records the whole decision, and the API
returns structured JSON.

**Appeal flow (narrative).** If a creator disputes a result, the platform calls
`POST /appeal` with the `submission_id` and the creator's written reasoning. The **appeals
handler** validates the submission, writes an appeal record *beside* the original decision in
the audit log, and flips that content's status to `under_review`. Re-classification is not
automated — a human moderator reviews from the appeal queue.

## 2. Detection Signals (Question 1)

The pipeline uses **two genuinely independent signals** — one *structural*, one *semantic*.
They fail in different ways, which is exactly why combining them beats either alone.

### Signal 1 — Stylometric heuristics (structural)

**What it measures.** Statistical regularities in the surface form of the writing, from three
sub-metrics computed in pure Python:
- **Sentence-length variance ("burstiness")** — how much sentence length varies.
- **Type-token ratio (TTR)** — vocabulary diversity: unique words ÷ total words.
- **Punctuation density** — punctuation marks per word (a proxy for rhythmic variety).

**Output shape.** A dict:
```json
{ "p_ai": 0.34, "self_confidence": 0.9,
  "metrics": { "sentence_len_variance": 41.2, "ttr": 0.68, "punct_density": 0.11 },
  "note": "high burstiness, diverse vocabulary → leans human" }
```
`p_ai` ∈ [0,1] is the signal's own estimate of P(AI). `self_confidence` ∈ [0,1] drops on
short texts where the statistics are unreliable (used to cap the fused result — see §3).

**Why it separates human from AI.** Human writing is bursty and uneven; instruction-tuned
LLMs regress to the mean — sentences cluster around a comfortable length, vocabulary is broad
but evenly deployed, punctuation is tidy. Low variance + smooth uniformity fingerprints AI.

**What it misses (blind spot).** It is blind to *meaning* and easily destabilized: short
texts make the statistics noise; intentionally uniform human forms (formal legal prose, a
constrained poem, technical docs) read "AI-uniform"; and it is trivially gameable by anyone
who varies sentence length on purpose, or by AI text lightly hand-edited for rhythm.

### Signal 2 — LLM classifier (semantic)

**What it measures.** A large model's holistic judgment of whether the text *reads* as
machine-generated — coherence that is too smooth, generic hedging, telltale phrasing, absence
of lived specificity. Captures the semantic gestalt stylometry can't see.

**Output shape.** The model is prompted to return strict JSON, which we parse to:
```json
{ "p_ai": 0.82, "rationale": "even, generic phrasing and hedged structure typical of AI",
  "available": true }
```
`available:false` when Groq errors/times out, so fusion can degrade gracefully (§3, §6).

**Why it separates human from AI.** LLMs recognize the register of other LLMs — the even,
slightly hollow "AI voice," missing specific detail, predictable structure. It catches AI
text that is *statistically* varied but semantically generic.

**What it misses (blind spot).** It is another AI judging AI (no ground truth, uncalibrated
confidence, can be confidently wrong); it is non-deterministic (same text can score
differently); it is prone to **false positives on polished or non-native-English human
writing** — the harm we most want to avoid; and it can be prompt-manipulated.

**Why the pair works.** The signals are decorrelated: stylometry is fooled by
statistically-varied-but-generic AI (the LLM catches it), and the LLM is fooled by polished
human prose (whose natural burstiness stylometry rewards). Requiring them to *agree* before a
strong call is the core safety mechanism (§3).

**Combination:** see §3 for the exact fusion formula.

*(Stretch — multi-modal: for `content_type: "image_metadata"`, Signal 1 is swapped for a
metadata-heuristics analyzer — missing/emptied EXIF, generator tags like "DALL·E"/"Midjourney",
suspiciously round dimensions — while Signal 2 runs the LLM on the caption/description. Same
fusion, same label logic. Documented in README when built.)*

## 3. Uncertainty Representation (Question 2)

**What I want the numbers to mean, decided before the math.** The system reports two numbers
because they answer two different questions:
- `ai_likelihood` ∈ [0,1] — *which direction and how strongly* the evidence points toward AI.
- `confidence` ∈ [0,100] — *how much you should trust the verdict.* 100 = sure, 0 = coin flip.

So **a confidence of 0.6 (60) means "we lean this way but wouldn't stake much on it"** — a
moderate call the label softens with hedged wording, not a firm verdict.

**Fusion (raw signal outputs → calibrated score):**
- Each signal returns `p_ai` ∈ [0,1].
- `ai_likelihood = 0.6 · p_ai_llm + 0.4 · p_ai_stylo`. The LLM is weighted higher because it
  reasons about meaning; stylometry is a cheaper, noisier corroborator.
- `agreement = 1 − |p_ai_llm − p_ai_stylo|` ∈ [0,1]. High when the signals concur.
- `confidence = round( |ai_likelihood − 0.5| · 2 · agreement · min_self_conf · 100 )`, where
  `min_self_conf` is the stylometry `self_confidence` (1.0 for normal-length text, lower for
  very short text). This means **three independent things drive confidence down**: a
  borderline combined score, disagreeing signals, and too-short input.
- If the LLM is unavailable, fusion uses stylometry alone and hard-caps `confidence` into the
  Low tier (single-signal results are never "high confidence").

**Thresholds — deliberately asymmetric** (false positives, i.e. calling a human's work AI,
are worse than false negatives on a writing platform, so `AI` is *hard to earn*):

| Condition | Verdict |
|---|---|
| `ai_likelihood ≥ 0.75` **and** `agreement ≥ 0.5` | `AI` |
| `ai_likelihood ≤ 0.35` | `HUMAN` |
| everything else (borderline, or signals disagree) | `UNCERTAIN` |

**Confidence tiers — decide which *wording* the label uses (not just a number):**

| Confidence | Tier | Label wording |
|---|---|---|
| ≥ 65 | High | committed verdict ("Likely AI-generated" / "Likely written by a person") |
| 35–64 | Moderate | softened verdict ("Leans toward…") |
| < 35 | Low | uncertain wording, regardless of direction |

**No binary flip at 0.5.** A `0.51` case (borderline, usually low agreement) → Low confidence
→ *Uncertain* label. A `0.95` case with both signals agreeing → High confidence → committed
*AI* label. A `0.78` case with moderate agreement → Moderate → *"Leans toward AI-generated."*
Three different labels across the range, not a switch.

**Validation plan (Milestone 4).** Feed a small labeled set — obvious ChatGPT output, obvious
raw human writing (typo-laden journal entries), and deliberately ambiguous cases (AI text
hand-edited for burstiness; polished ESL human prose). Confirm the obvious cases clear the
high-confidence thresholds and the ambiguous ones land `UNCERTAIN` with confidence < 35.
Re-submit identical text several times to observe LLM non-determinism and confirm it doesn't
flip a high-confidence verdict.

## 4. Transparency Label Design (Question 3)

Plain language, no jargon (no "classifier," "logit," "score vector"), and the *wording*
changes with confidence, not just the number. Exact strings the reader sees:

**High-confidence AI**
> 🤖 **Likely AI-generated.** Our checks strongly suggest this text was produced by an AI
> tool. Both how it's written (sentence rhythm and word variety) and how it reads overall
> point the same way. *Confidence: high (91%).* This is an automated assessment, not a
> certainty — if you wrote this and disagree, you can appeal.

**High-confidence human**
> ✍️ **Likely written by a person.** Our checks found the natural variety in rhythm and word
> choice we'd expect from a human writer, and nothing that points to AI generation.
> *Confidence: high (88%).* No automated check is perfect, but this reads as human work.

**Uncertain**
> ❓ **We're not sure.** Our checks came back mixed or borderline, so we're not putting a
> label on this one. *Confidence: low (38%).* We'd rather say "we don't know" than risk
> wrongly calling a person's work AI-generated. A human reviewer can take a closer look if
> the creator requests it.

**Moderate tier (softened, reuses direction).** e.g. *"🤖 Leans toward AI-generated. Some
signs point to AI authorship, but this isn't a firm call. Confidence: moderate (58%). If you
wrote this and disagree, you can appeal."* — and the human-leaning mirror image.

*(Reviewed and kept: the uncertain variant leads with "We're not sure" and states the
false-positive rationale in plain terms, which is the wording a non-technical reader
understood most clearly in an informal read-through.)*

## 5. Appeals Workflow (Question 4)

**Who can appeal.** The creator of the content (the platform passes the appeal on their
behalf, identified by `submission_id`; `creator_id` from the original submission ties the
appeal to its author). Any classification can be appealed, but `AI` and `UNCERTAIN` verdicts
are the realistic cases.

**What they provide.** `submission_id` plus **`creator_reasoning`** — free-text explanation of
why they believe the classification is wrong (e.g. "English is my second language; this is my
own personal essay"). Reasoning is required; an empty appeal is rejected (`400`).

**What the system does on receipt.**
1. Validate the `submission_id` exists (`404` if not).
2. Write a new **appeal entry** to the audit log, linked to the original decision's
   `submission_id`, containing: `appeal_id`, `creator_reasoning`, `logged_at` timestamp, and a
   snapshot of the original verdict/confidence being contested.
3. Update the content's **status → `under_review`** (from `classified`).
4. Return `{ appeal_id, submission_id, status: "under_review", logged_at }`.

**Automated re-classification is intentionally out of scope** — an appeal escalates to a
human, it does not silently re-run the detector.

**What a human reviewer sees in the appeal queue.** `GET /log?status=under_review` (and
`GET /submission/<id>`) surface, for each open appeal, a single joined view:
- the **original submission** — content excerpt, `verdict`, `ai_likelihood`, `confidence`,
  both individual signal scores + the LLM rationale, the label shown, original timestamp; and
- the **appeal** — `creator_reasoning`, `appeal_id`, `logged_at`, current status.

That gives a moderator everything needed to overturn or uphold in one place, without re-running
anything.

## 6. Anticipated Edge Cases (Question 5)

1. **Very short submissions (a haiku, a one-line epigraph, a tweet-length excerpt).**
   Stylometric variance is meaningless below a minimum length — three sentences can't have a
   trustworthy "variance." *Handling:* below a word/sentence threshold, stylometry returns a
   low `self_confidence`; fusion multiplies it in, forcing the result into the Low-confidence /
   `UNCERTAIN` band. We never issue a high-confidence verdict on too little text.

2. **A poem built on heavy repetition and simple vocabulary** (e.g. a villanelle, or a
   deliberately plain minimalist piece). Low TTR + low sentence-length variance look exactly
   like the "uniform" AI fingerprint, so **stylometry false-positives toward AI**. *Handling:*
   the LLM signal typically reads genuine poetic craft as human, the two signals disagree,
   `agreement` drops, and the fused result lands `UNCERTAIN` rather than falsely accusing the
   poet. The asymmetric 0.75 AI threshold is the backstop.

3. **Polished non-native-English human writing.** Both signals can lean AI at once (clean,
   formal register). *Handling:* the high `AI` threshold (0.75) + agreement requirement keeps
   these `UNCERTAIN`, and the appeal path gives the creator recourse.

4. **Groq API down / times out.** *Handling:* pipeline degrades to stylometry alone, hard-caps
   confidence into the Low tier, and the audit log records `llm.available: false`.

## 7. Rate Limiting

Applied to `POST /submit`. Limits tied to realistic writing-platform behavior:
- **10 submissions / minute / client** — a real creator publishes a handful of pieces in a
  sitting; 10/min comfortably covers edit-and-resubmit while staying far below what a flooding
  script needs (each call costs a Groq request).
- **100 submissions / hour / client** — a hard ceiling that stops sustained abuse (e.g.
  probing the detector to reverse-engineer thresholds) without ever inconveniencing a real
  writer.

Exceeding a limit returns **HTTP 429** with a JSON error and `Retry-After`. Final numbers +
the reasoning table are restated in the README.

## 8. API Surface

| Method & path | Accepts | Returns |
|---|---|---|
| `POST /submit` | `{ "content": str, "creator_id"?: str, "content_type"?: "text"\|"image_metadata" }` | `{ submission_id, verdict, ai_likelihood, confidence, signals: { llm, stylometry }, label, status, timestamp }` |
| `POST /appeal` | `{ "submission_id": str, "creator_reasoning": str }` | `{ appeal_id, submission_id, status: "under_review", logged_at }` |
| `GET /log` | optional `?limit=`, `?status=` | structured JSON array of audit entries (decisions + appeals) |
| `GET /submission/<id>` | — | original record + any appeal + current status |
| `GET /analytics` *(stretch)* | — | `{ total, ai_vs_human_ratio, uncertain_rate, appeal_rate }` |
| `GET /` | — | health check |

Error contract: `400` bad/missing input, `404` unknown `submission_id`, `429` rate-limited,
`500` upstream failure with graceful degradation to the stylometric signal only.
